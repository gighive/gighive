# Problem: QR Code Scan Does Not Open iOS App (Universal Links Not Firing)

## Summary

When a guest scans the QR code generated by the admin Guest QR Upload page, the iPhone Camera app opens the URL in a browser instead of launching the GigHive app directly. The feature relies on iOS Universal Links, which require a precise multi-layer configuration spanning the server, Apple's CDN, the iOS app's code signing, the Xcode project, and the device itself. Any single misconfiguration silently prevents the redirect.

This document records every problem encountered during initial setup and serves as the definitive configuration reference for the feature going forward.

---

## Impact

- Guests with QR links were directed to the web-based upload form in Safari or Chrome instead of the native iOS upload flow.
- The failure was silent — no error message on device, no server-side log entry. Console.app on Mac with `swcd` filter was required to identify the root cause.
- All problems were ultimately resolved. The feature works correctly when the full checklist below is satisfied.

---

## Symptoms

- iPhone Camera detects the QR code, shows the domain badge (e.g. `dev.gighive.app`), but taps open a browser.
- Tapping the domain badge opens Safari, but no "Open in GigHive" banner appears at the top.
- On iOS 26 with Chrome as the default browser, Camera shows an "Open in Chrome" badge instead of "Open in GigHive".

---

## Problems Encountered (Chronological)

### 1 — iOS 14 API incompatibilities in `GuestUploadView.swift`

`@Environment(\.dismiss)` and `.task(id:)` are iOS 15+ APIs. The minimum deployment target is iOS 14.

**Fix:** Replace with `@Environment(\.presentationMode)` / `.wrappedValue.dismiss()` and `.onAppear` + `.onChange(of:)` respectively.

---

### 2 — `.onOpenURL` placed on `WindowGroup` (Scene modifier, not View modifier)

`.onOpenURL` is a `View` modifier. Attaching it directly to `WindowGroup` (a `Scene`) compiles but does not fire when a Universal Link is opened.

**Fix:** Wrap the navigation content in a `Group {}` and attach `.onOpenURL` to the `Group`.

```swift
WindowGroup {
    Group {
        if #available(iOS 16.0, *) {
            NavigationStack { SplashView() }
        } else {
            NavigationView { SplashView() }
        }
    }
    .onOpenURL { url in
        // parse url and update guestSession
    }
    .environmentObject(guestSession)
}
```

---

### 3 — Wrong bundle ID in all AASA `appIDs` entries

The spec documentation and all Ansible group_vars were written with `com.gighive.GigHive` as the bundle identifier. The actual registered Apple App ID (visible in Xcode → Signing & Capabilities and confirmed via `codesign`) is `app.gighive.GigHive`.

iOS performs an exact string comparison of the AASA `appIDs` against the signed app's `application-identifier` entitlement. The mismatch caused Universal Links to be silently rejected.

**Diagnosis:**
```bash
codesign -d --entitlements - /path/to/GigHive.app
# Shows: application-identifier = WB7D4FC7XU.app.gighive.GigHive
```

**Fix:** Update `qr_aasa_app_id` in all three group_vars and redeploy:
```yaml
qr_aasa_app_id: "WB7D4FC7XU.app.gighive.GigHive"   # was: WB7D4FC7XU.com.gighive.GigHive
```

Files updated:
- `ansible/inventories/group_vars/gighive/gighive.yml`
- `ansible/inventories/group_vars/gighive2/gighive2.yml`
- `ansible/inventories/group_vars/prod/prod.yml`

Also corrected in `GigHive/project.yml`:
```yaml
PRODUCT_BUNDLE_IDENTIFIER: app.gighive.GigHive   # was: com.gighive.GigHive
```

---

### 4 — Apple's CDN had stale AASA cached with the wrong app ID

Apple's CDN (`app-site-association.cdn-apple.com`) had previously crawled `dev.gighive.app` and cached the AASA with the incorrect `com.gighive.GigHive` app ID. Even after correcting the server-side AASA, iOS continued to receive the wrong app ID from Apple's CDN cache.

**Diagnosis:**
```bash
curl -s https://app-site-association.cdn-apple.com/a/v1/dev.gighive.app
# Returned stale: "WB7D4FC7XU.com.gighive.GigHive"
```

**Fix:** No manual flush is available. Apple's CDN re-crawls automatically within approximately 24 hours of detecting a change. Once the CDN updates, Universal Links begin working without any further app or device changes.

---

### 5 — `?mode=developer` does not bypass Apple's CDN in iOS 26

In iOS 15/16, `?mode=developer` in an `applinks:` entitlement caused `swcd` to fetch the AASA directly from the origin server at install time, bypassing Apple's CDN entirely. In iOS 26, `swcd` still checks Apple's CDN first even for `?mode=developer` domains. If the CDN returns a 404 or a mismatched AASA, the developer-mode direct fetch is not attempted.

**Evidence from Console.app (`swcd` filter):**
```
Request for '<private>' for task AASA-... { domain: gi...app, bytes: 0, route: cdn }
denied because the CDN
response_status=404
```

**Implication:** `?mode=developer` alone is insufficient on iOS 26 if Apple's CDN does not have a matching AASA entry. The CDN must also have the correct AASA cached.

---

### 6 — `Associated Domains Development` toggle off — swcd silently skips all `?mode=developer` domains

**Settings → Developer → Associated Domains Development** is a separate toggle from the general **Settings → Privacy & Security → Developer Mode**. When this toggle is OFF, `swcd` silently discards every `applinks:` entitlement that has `?mode=developer` and logs:

```
swcd    Developer mode enabled: NO
```

This log message is misleading: "Developer Mode" in Privacy & Security can be ON while swcd still reports NO, because swcd is checking the *Associated Domains Development* toggle, not the general Developer Mode setting.

**Evidence:** `dev.gighive.app` never appeared anywhere in the swcd AASA batch fetch log; after enabling the toggle, the log immediately showed:
```
swcd    Developer mode enabled: YES
swcd    Beginning data task AASA-... { domain: de….gi….app?mode=developer, bytes: 0, route: .wk }
```

**Fix:** On device: Settings → Developer → Associated Domains Development → ON. Then delete and reinstall the app.

---

### 7 — `gighive.app` production CDN returns 404

The production domain `gighive.app` has no AASA cached by Apple's CDN (`curl -s https://app-site-association.cdn-apple.com/a/v1/gighive.app` → `Not Found`). Having this domain in the entitlement without a matching CDN entry causes the entire entitlement set to fail validation silently on iOS 26. **Fix:** Remove `applinks:gighive.app` from the entitlements until the AASA is deployed to the production server and Apple's CDN has crawled it.

---

### 8 — Chrome as default browser prevents Universal Links in Camera

When Chrome (or any non-Safari browser) is set as the iPhone's default browser, the Camera app routes QR code URLs directly to that browser without going through the iOS Universal Links subsystem. The Universal Link never fires regardless of how correctly the AASA and entitlements are configured.

**Fix (for development):** Set Safari as the default browser while testing. For end-users in production, this is a known iOS platform limitation — guests using Chrome as default will land on the web fallback form, which is acceptable behavior.

---

### 9 — Admin page missing `Cache-Control: no-store` headers (Cloudflare caching risk)

`event_qr.php` emitted no cache headers. Cloudflare could cache the admin page, causing stale QR codes or frozen status messages (consistent with the behavior observed on other admin pages).

**Fix:** Added to the top of `event_qr.php` after `session_start()`:
```php
header('Cache-Control: no-store, no-cache, must-revalidate');
header('Pragma: no-cache');
```

---

### 10 — Internal server TLS certificate lacks `CA:TRUE` — swcd rejects with `TrustResultType=5`

When using `?mode=developer` against an internal server (e.g. `gighive2.gighive.internal`), `swcd` fetches the AASA directly from the server and validates its TLS certificate using `SWCSecurityGuard`. Even if the server's self-signed certificate is installed on the device and toggled ON in **Certificate Trust Settings**, swcd will still fail if the certificate does not have `basicConstraints = CA:TRUE`.

**Symptom in Console.app swcd logs:**
```
Trust evaluate failure: [leaf AnchorTrusted]
Failed to verify server trust for task AASA-... { domain: gi….gi….internal?mode=developer }: 
  Error Domain=SWCErrorDomain Code=100 "Disallowed trust result type."
  UserInfo={Function=-[SWCSecurityGuard verifyTrust:allowInstalledRootCertificates:error:], TrustResultType=5}
```

`TrustResultType=5` is `kSecTrustResultRecoverableTrustFailure` — the cert is present and installed but trustd cannot verify a chain to a CA anchor. `SWCSecurityGuard.allowInstalledRootCertificates:` rejects this result type unconditionally.

**Fix:** The internal server's TLS certificate must be generated with `basicConstraints = critical, CA:TRUE, pathlen:0` so that iOS/trustd treats it as a root CA anchor and returns `kSecTrustResultUnspecified` or `kSecTrustResultProceed` — both of which swcd accepts. In the GigHive Ansible setup, add to `openssl_san.cnf.j2`:
```
basicConstraints = critical, CA:TRUE, pathlen:0
keyUsage = critical, digitalSignature, keyEncipherment, keyCertSign
```
Then redeploy (Ansible playbook regenerates the cert via the container entrypoint). Remove the old profile from the device, install the new `.crt`, enable full trust, and do a clean reinstall of the app.

**Verify the deployed cert has CA:TRUE:**
```bash
echo | openssl s_client -connect gighive2.gighive.internal:443 2>/dev/null | openssl x509 -noout -text | grep -A2 "Basic Constraints"
# Must show: CA:TRUE, pathlen:0
```

## Full Configuration Checklist

All of the following must be satisfied for Universal Links to fire and open the app.

| # | Layer | Requirement | How to verify |
|---|-------|-------------|---------------|
| 1 | Server | AASA file exists at `https://<domain>/.well-known/apple-app-site-association` | `curl -s https://dev.gighive.app/.well-known/apple-app-site-association` |
| 2 | Server | AASA `appIDs` value exactly matches `<TeamID>.<BundleID>` of the signed app | Compare `codesign` output `application-identifier` against AASA `appIDs` |
| 3 | Server | AASA served with `Content-Type: application/json` | `curl -sI https://dev.gighive.app/.well-known/apple-app-site-association` |
| 4 | Server | AASA served over HTTPS with a certificate trusted by the device | No browser warning when accessing the AASA URL directly on iPhone |
| 5 | Server | QR code URL uses the exact domain registered in the app's entitlements | Admin page accessed via `https://dev.gighive.app/admin/event_qr.php` so `$_SERVER['HTTP_HOST']` produces a matching token URL |
| 6 | Apple CDN | AASA for the domain is crawled and cached by `app-site-association.cdn-apple.com` with correct `appIDs` | `curl -s https://app-site-association.cdn-apple.com/a/v1/<domain>` |
| 7 | Xcode project | Associated Domains capability added in Signing & Capabilities tab (not just in entitlements plist) | Xcode → GigHive target → Signing & Capabilities → Associated Domains section visible |
| 8 | Xcode project | `GigHive.entitlements` contains correct `applinks:` entries | Read `Configs/GigHive.entitlements` |
| 9 | Xcode project | `project.yml` `PRODUCT_BUNDLE_IDENTIFIER` matches the actual registered Apple App ID | `app.gighive.GigHive` in `project.yml` and confirmed in Xcode General tab |
| 10 | iOS app code | `.onOpenURL` attached to a `View` (e.g. `Group`), not to `WindowGroup` (a `Scene`) | `GigHiveApp.swift` — `.onOpenURL` is on `Group {}` wrapper |
| 11 | iOS app code | All SwiftUI APIs are iOS 14 compatible (no `.dismiss`, no `.task(id:)`) | `GuestUploadView.swift` uses `presentationMode` and `.onAppear`/`.onChange` |
| 12 | iOS app code | `onOpenURL` parses `/upload/<token>` path and updates `GuestUploadSession.rawToken` | Trace `GigHiveApp.swift` URL parsing logic |
| 13 | Device | Developer Mode enabled (iOS 16+) | Settings → Privacy & Security → Developer Mode → ON |
| 14 | Device | App installed via Xcode debug build (not TestFlight/Ad-hoc) when using `?mode=developer` | Hit ▶ in Xcode with device selected; not distributed via other means |
| 15 | Device | App fully deleted before reinstalling when changing entitlements or AASA | Long-press icon → Remove App → Delete App; then reinstall via Xcode |
| 16 | Device | Safari is the default browser (Chrome as default bypasses Universal Links in Camera) | Settings → Apps → Default Apps → Browser → Safari |
| 17 | Device | **Associated Domains Development** toggle enabled (separate from Developer Mode) | Settings → Developer → Associated Domains Development → ON. Without this, swcd logs `Developer mode enabled: NO` and silently skips all `?mode=developer` domains |
| 18 | Server | Production domain AASA cached on Apple CDN if `applinks:<prod-domain>` (no `?mode=developer`) is in entitlements | `curl -s https://app-site-association.cdn-apple.com/a/v1/gighive.app` — must not return `Not Found`. Remove the entitlement entry until the CDN has it. |
| 19 | Device / Server | Internal server TLS cert has `CA:TRUE` — required for swcd to trust it as a root anchor | `echo \| openssl s_client -connect <host>:443 2>/dev/null \| openssl x509 -noout -text \| grep -A2 "Basic Constraints"` — must show `CA:TRUE`. Without this, swcd logs `TrustResultType=5` and rejects the AASA fetch even if the cert is installed and enabled in Certificate Trust Settings. |

## Debugging Tools

### Check signed entitlements in the built binary
```bash
codesign -d --entitlements - \
  ~/Library/Developer/Xcode/DerivedData/<DerivedData>/Build/Products/Debug-iphoneos/GigHive.app
```
Confirms `com.apple.developer.associated-domains` is present and lists the correct domains.

### Check Apple CDN cache for a domain
```bash
curl -s https://app-site-association.cdn-apple.com/a/v1/<domain>
```
Returns the AASA as Apple's CDN has it. If `appIDs` is wrong or the response is `Not Found`, Universal Links will not fire regardless of entitlements or `?mode=developer`.

### Check AASA headers (Content-Type must be `application/json`)
```bash
curl -sI https://<domain>/.well-known/apple-app-site-association
```

### Read `swcd` logs on device during app install (definitive diagnosis)
1. Connect iPhone to Mac
2. Open Console.app → select the iPhone → Start Streaming
3. Filter: `swcd`
4. Delete app from iPhone, reinstall via Xcode ▶
5. Look for: `route: cdn`, `response_status`, `denied because the CDN`, `attribution: developer`

### Test Universal Links on iOS Simulator (bypasses Apple CDN entirely)
```bash
xcrun simctl openurl <simulator-udid> \
  "https://dev.gighive.app/upload/<token>"
```
Use this to validate the app-side `onOpenURL` handler without CDN dependencies.

### Direct device URL dispatch (physical device, bypasses Camera)
```bash
xcrun devicectl device process launch \
  --device <device-name-or-udid> \
  app.gighive.GigHive \
  --url "https://dev.gighive.app/upload/<token>"
```

| Problem | Fix |
|---------|-----|
| iOS 14 API incompatibilities | Replaced `dismiss` / `.task(id:)` with `presentationMode` / `.onAppear` + `.onChange` |
| `.onOpenURL` on Scene | Moved to `Group {}` wrapper inside `WindowGroup` |
| Wrong bundle ID in AASA | Updated `qr_aasa_app_id` to `WB7D4FC7XU.app.gighive.GigHive` in all group_vars; corrected `project.yml` |

## Preventative Actions

- Always verify `codesign` output and Apple CDN before declaring Universal Links working.
- When changing the AASA, allow up to 24 hours for Apple's CDN to re-crawl before concluding Universal Links are broken.
- Access the admin QR page via the same public domain used in the app's entitlements (e.g. `dev.gighive.app`) so the generated token URL domain is always correct.
- Keep `project.yml` `PRODUCT_BUNDLE_IDENTIFIER` and the AASA `appIDs` in sync; the mismatch is invisible at compile time and only surfaces at runtime via Console.app `swcd` logs.
- Add a smoke test that compares `curl https://app-site-association.cdn-apple.com/a/v1/<domain>` against the expected `appIDs` value in CI or during deployment verification.
