# Swagger PHP Upgrade Plan (4.11.1 → 5.7.x)

## Current Situation

Current installed version:

```text
zircote/swagger-php 4.11.1
```

Composer currently allows:

```json
"zircote/swagger-php": "^4.0"
```

Dependabot opened a PR to upgrade to:

```text
5.7.7
```

However, the PR is stale and contains merge conflicts because it was left open while additional commits were made to `master`.

---

# Recommended Approach

Do NOT try to fix the old Dependabot PR.

Instead:

1. Close the old PR
2. Perform the upgrade manually on master
3. Test the application
4. Push and create a new clean PR

---

# Step-by-Step TODO List

## 1. Close the Old Dependabot PR

Close the old Swagger PR in GitHub.

Reason:
- Merge conflicts
- Hundreds of changed files
- Stale branch

---

## 2. Update composer.json Only

File:

```text
ansible/roles/docker/files/apache/webroot/composer.json
```

Change this:

```json
"zircote/swagger-php": "^4.0"
```

To this:

```json
"zircote/swagger-php": "^5.7"
```

Do NOT manually edit `composer.lock` — it is auto-generated by Composer in the next step.

---

## 3. Run Composer Update

From the webroot directory:

```bash
composer update zircote/swagger-php
```

This updates `composer.lock` and the `vendor/` directory. (`composer.json` was already updated in step 2.)

If Composer reports dependency conflicts, stop and investigate before proceeding. Do NOT force the upgrade.

---

## 4. Regenerate the OpenAPI Spec

From the webroot directory:

```bash
composer run openapi
```

This runs:

```bash
vendor/bin/openapi src/ --output docs/openapi.yaml --format yaml
```

Check for PHP errors or annotation parsing warnings in the output.

---

## 5. Inspect the Generated Spec Diff

```bash
git diff docs/openapi.yaml
```

Expected changes:
- Version header changes from `openapi: 3.0.0` to `openapi: 3.1.0`
- `nullable: true` fields re-rendered as type unions (e.g. `type: [string, null]`) — this is correct OpenAPI 3.1 format

If any previously-nullable fields have **disappeared entirely** from the output, proceed to step 6. Otherwise skip to step 7.

---

## 6. (Conditional) Fix nullable Annotations

Only needed if step 5 reveals nullable fields dropped from output.

Files to edit:

```text
ansible/roles/docker/files/apache/webroot/src/OpenApi.php
ansible/roles/docker/files/apache/webroot/src/Controllers/UploadController.php
```

Replace occurrences of:

```php
nullable: true
```

With the OpenAPI 3.1 union form:

```php
type: ['string', 'null']   // or ['integer', 'null'], ['number', 'null'] etc. to match existing type
```

Re-run step 4 after editing and re-inspect the diff.

---

## 7. Test the Application

At minimum:
- Start the application
- Verify APIs still work
- Verify Swagger UI documentation still loads correctly

---

## 8. Commit the Upgrade

Since `vendor/` is tracked in git, the upgrade replaces all swagger-php v4 library files with v5 and adds new transitive dependencies (`nikic/php-parser`, `phpstan/phpdoc-parser`). Stage everything under webroot at once from the repo root:

```bash
git add ansible/roles/docker/files/apache/webroot/
```

Then commit:

```bash
git commit -m "Upgrade swagger-php to 5.7"
```

Note: `CHANGELOG.md` will also appear modified — update it with an entry for this upgrade and commit separately.

---

## 9. Push and Create PR

```bash
git push origin master
```

Create a new PR in GitHub if required, or merge directly.

---

# Important Notes

## If Composer Update Fails

Stop and investigate before proceeding.

Possible causes:
- PHP version incompatibility
- Other Composer dependency conflicts
- Swagger annotation compatibility changes

Do NOT force the upgrade if Composer reports dependency issues.

## OpenAPI 3.1 Output

swagger-php v5 outputs OpenAPI 3.1 (was 3.0.0 in v4). Verify that any consumers of `docs/openapi.yaml` (Swagger UI, Postman, etc.) support OpenAPI 3.1 before merging.

## Annotation Syntax

The codebase uses PHP 8 native attributes (`#[OA\...]`) exclusively — no docblock-style `@OA\...` annotations. This means the primary v5 breaking change (dropped docblock support) does not affect this codebase.
