# Manual Tagging for Assets

Date: 2026-05-01

---

## Rationale and Go-Forward Path

The `tags` and `taggings` schema was introduced as part of the AI Video Tagger feature, but the tables are source-agnostic — `source ENUM('ai','human')` means human and AI tags are first-class equals in the same structure.

### Why manual tagging comes first

The AI worker requires an OpenAI API key, Docker infrastructure changes, and significant first-run download overhead (~500–700 MB). It is an opt-in add-on that only benefits deployments that explicitly enable it. Manual tagging, by contrast:

- Works for **every deployment** with zero additional setup
- Delivers immediate value — admins can categorize, find, and filter assets today
- Builds a tag vocabulary that AI tags will slot into naturally when enabled later
- Validates the tag UX before committing to AI-generated content

### Go-forward path

| Phase | What | Gate |
|---|---|---|
| **Phase 1 (this doc)** | Manual tagging — admin can add/delete/browse human-sourced tags on any asset | No external dependencies |
| **Phase 2** | AI auto-tagging on full VM deployments (gighive/gighive2) | `ai_worker_enabled: true` + `OPENAI_API_KEY` in secrets.yml |
| **Phase 3** | AI worker in the one-shot bundle (see `feature_ai_video_tagger_osb.md`) | Phase 2 proven stable; OSB assembly changes complete |

Human and AI tags coexist in the same tables throughout all phases. A tag added manually in Phase 1 and the same tag generated by AI in Phase 2 merge to a single `tags` row — the `UNIQUE KEY uq_taggings_tag_target` prevents duplicates and the `source` column tracks origin.

## Step 1 — Link to per-asset tag page from the media library

**What already exists**: The Tags column JS in `list.php` already renders a **"Tag this video"** CTA button (line ~618) linking to `media_tags.php?asset_id=N` when an asset has zero tags. The zero-tags case is already handled.

**The real gap**: Once an asset *has* tags, the chips link to `tag_browser.php?namespace=...&name=...` (the cross-asset browser by tag name). There is no path back to the per-asset management page for an already-tagged asset — so an admin cannot add more tags or remove existing ones from the list view.

**Change**: `src/Views/media/list.php` (JS `loadTagChips` function)

In the branch that renders chips for tagged assets, append a small **"✎"** manage link after the chip wrap:

```js
const manageLink = document.createElement('a');
manageLink.href = '/db/media_tags.php?asset_id=' + aid;
manageLink.textContent = '✎';
manageLink.title = 'Manage tags';
manageLink.style.cssText = 'font-size:.72rem;color:#60a5fa;margin-left:4px;text-decoration:none;';
container.appendChild(manageLink);
```

This gives both paths from the list view:
- **No tags** → "Tag this video" button (already exists)
- **Has tags** → chips (tag browser) + "✎" link (per-asset management)

**Scope**: `src/Views/media/list.php` JS only. No PHP or controller changes.

## Step 2 — Allow custom namespaces in the "Add Manual Tag" form

**Problem**: The namespace `<select>` in `db/media_tags.php` is a hardcoded 4-option dropdown (`scene`, `object`, `activity`, `person_role`). There is no way to add a tag in a different namespace.

**Change**: `db/media_tags.php`

Add an `other...` option to the `<select>`. When selected, reveal a text `<input>` for the custom namespace. The JS `addTagBtn` click handler already reads `document.getElementById('newNs').value`; update it to read from the text input instead when "other" is selected.

```html
<select id="newNs">
  <option value="scene">scene</option>
  <option value="object">object</option>
  <option value="activity">activity</option>
  <option value="person_role">person_role</option>
  <option value="__custom__">other…</option>
</select>
<input type="text" id="customNs" placeholder="custom namespace"
       style="display:none; width:120px;">
```

These two concerns (custom-ns toggle + datalist fetch) must be handled in a **single merged `change` listener** — not two separate ones — to avoid ordering issues and the async fetch racing the toggle:

```js
document.getElementById('newNs').addEventListener('change', async function() {
    const isCustom = this.value === '__custom__';
    document.getElementById('customNs').style.display = isCustom ? '' : 'none';

    const dl = document.getElementById('tagNameList');
    dl.innerHTML = '';  // clear hints on every namespace switch

    if (isCustom) return;  // no namespace known yet, leave datalist empty

    try {
        const r = await fetch('/api/tags.php?namespace=' + encodeURIComponent(this.value));
        const d = await r.json();
        (d.tags || []).forEach(t => {
            const opt = document.createElement('option');
            opt.value = t.name;
            dl.appendChild(opt);
        });
    } catch (_) { /* non-fatal — hints just won't appear */ }
});
```

In the `addTagBtn` click handler, resolve `ns` as:
```js
const nsEl = document.getElementById('newNs');
const ns = nsEl.value === '__custom__'
    ? document.getElementById('customNs').value.trim().toLowerCase().replace(/[^a-z0-9_]/g,'')
    : nsEl.value;
if (!ns) { setMsg('Enter a namespace.', false); return; }
```

Add `list="tagNameList"` to the `<input type="text" id="newName">` and a `<datalist id="tagNameList">` alongside it. When the table is empty or "other…" is selected, the datalist is empty — no hints, no error. Hints grow automatically as tags accumulate. Note: custom namespaces get no datalist hints (namespace unknown until the user types it).

**Scope**: `db/media_tags.php` only. `api/taggings.php` already accepts any namespace string; `api/tags.php` already returns tags by namespace.

## Step 4 — Add tag filter to the media library search

**Problem**: The librarian search in `AssetRepository` supports `file_type`, `source_relpath`, `duration_seconds`, `media_info` — but not tags. Users cannot filter the library to "show all assets tagged `outdoor_concert`".

**Changes**:

### `src/Repositories/AssetRepository.php`

Add `tag` as a supported filter key in `buildLibrarianFilters()`. The SQL approach is an `EXISTS` subquery to avoid row duplication from multi-tag JOINs:

```php
'tag' => [
    'sql'        => 'EXISTS (SELECT 1 FROM taggings tg2 JOIN tags t2 ON t2.id=tg2.tag_id WHERE tg2.target_type=\'asset\' AND tg2.target_id=a.asset_id AND LOWER(t2.name) LIKE LOWER(%s))',
    'param_base' => 'tag',
],
```

This reuses the existing OR/AND/NOT filter logic without any structural changes to `buildLibrarianFilters`.

### `src/Views/media/list.php`

Add a "Tags" text input to the search form alongside the existing `file_type`, `source_relpath` etc. inputs. The form already POSTs/GETs these as query parameters; no controller change needed.

### `src/Controllers/MediaController.php`

Add `'tag'` to the `getFiltersFromRequest()` allowed keys (or wherever filters are extracted from `$_GET`).

**Scope**: `AssetRepository.php`, `list.php` (search form), `MediaController.php` (filter key allow-list).

## Roadmap

### Tag time-range (location within asset)

The `taggings` table already has `start_seconds` and `end_seconds` columns (nullable `FLOAT`), populated by the AI worker for frame-level tags. Manual tags currently leave these `NULL`, meaning the tag applies to the whole asset.

**Proposed UX** (Add Manual Tag form in `db/media_tags.php`):
- Add optional **Start** and **End** time inputs (e.g. `<input type="number" step="0.1" min="0" placeholder="hh:mm:ss or seconds">`)
- Leave both blank → tag applies to whole asset (current behavior, unchanged)
- Populate one or both → stored as `start_seconds` / `end_seconds` in the `taggings` row
- Display time range on existing chips in the tag list (e.g. `outdoor_concert · 0:12–0:45`)

**API change required**: `POST /api/taggings.php` already accepts `start_seconds` and `end_seconds` in the JSON body; the DB write path supports them. Only the form UI and chip display need to change.

**Scope**: `db/media_tags.php` form + chip rendering only — no schema or API changes needed.
